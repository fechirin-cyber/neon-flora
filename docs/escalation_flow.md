# NEON FLORA 監査・エスカレーションフロー

**バージョン**: 1.0
**最終更新**: 2026-03-01
**承認状態**: プロデューサー確認待ち

---

## 1. 基本原則

**全ての成果物は、ワーカー → ディレクター → プロデューサーの順序で上がる。例外なし。**

```
┌──────────┐    ┌──────────────┐    ┌──────────────┐
│ ワーカー  │───→│ ディレクター  │───→│ プロデューサー │
│ (実作業)  │    │ (品質監査)    │    │ (最終決定)    │
└──────────┘    └──────────────┘    └──────────────┘
      ↑               │ BLOCK
      └───────────────┘ (差し戻し)
```

### 禁止事項

1. ワーカーの成果物をディレクター監査なしでプロデューサーに報告すること
2. フェーズ要件が未達成のままプロデューサーに報告すること
3. ディレクターがテストプレイ成果物を確認せずに APPROVE すること
4. ワーカーの報告を鵜呑みにして監査を省略すること
5. BLOCK が出ているのにプロデューサーに上げること

## 2. チーム体制と監査ライン

### 2.1 組織図

```
プロデューサー（ユーザー）
  │
  ├─ リードプログラマー（Claude本体）── 実装統括、全体調整
  │
  ├─ ゲームデザインリード ←── プランナー
  │
  ├─ テクニカルディレクター ←── テクニカルリード、デバッガー
  │
  ├─ アートディレクター ←── アーティスト、VFXデザイナー、UIデザイナー
  │
  ├─ サウンドディレクター ←── サウンドデザイナー
  │
  └─ QAディレクター ←── QAリード（横断監査）
```

### 2.2 監査ライン一覧

| ディレクター | 監査対象ワーカー | 判定権限 |
|---|---|---|
| ゲームデザインリード | プランナー | 企画の市場品質・コンセプト適合・ゲーム体験 |
| テクニカルディレクター | テクニカルリード, デバッガー | 安定性・パフォーマンス・アーキテクチャ |
| アートディレクター | アーティスト, VFX, UIデザイナー | ビジュアル品質・世界観統一・視認性 |
| サウンドディレクター | サウンドデザイナー | 音響品質・世界観統一・体験向上 |
| QAディレクター | 全ワーカー（横断） | テスト網羅性・品質基準・最終出荷判定 |

## 3. 判定基準

### 3.1 ディレクター判定（3段階）

| 判定 | 意味 | アクション |
|---|---|---|
| **APPROVE** | フェーズ要件充足、品質基準達成 | プロデューサーに報告して良い |
| **CONDITIONAL** | 軽微な条件付きで承認 | 条件を明記してプロデューサーに報告（条件の対応はプロデューサー判断） |
| **BLOCK** | フェーズ要件未達成または品質基準未達 | **プロデューサーに報告しない**。修正指示を出して差し戻し |

### 3.2 BLOCK 時の対応フロー

```
1. ディレクターが BLOCK 判定 + 修正指示を出す
2. リードプログラマーが修正を実施（またはワーカーに委任）
3. 再ビルド → 再テストプレイ
4. ワーカーが再レビュー
5. ディレクターが再監査
6. APPROVE になるまで繰り返す
7. 全ディレクター APPROVE → プロデューサーに報告
```

## 4. フェーズレビュー手順

### 4.1 全体フロー

```
Phase 1: テストプレイ実行
  ↓
Phase 2: ワーカー6名が並列レビュー
  ↓
Phase 3: ディレクター5名が並列監査
  ↓
Phase 4: 判定
  ├─ 全 APPROVE → Phase 5（プロデューサー報告）
  ├─ CONDITIONAL のみ → Phase 5（条件付き報告）
  └─ BLOCK あり → 修正 → Phase 1 に戻る
  ↓
Phase 5: レビュー結果統合 + プロデューサー報告
```

### 4.2 Phase 1: テストプレイ実行

- 最新コードで EXE をビルド
- テストスクリプトを実行
- 成果物: スクリーンショット、debug_state.json、操作ログ

### 4.3 Phase 2: ワーカーレビュー（6名並列）

| # | ワーカー | レビュー観点 |
|---|---|---|
| 1 | プランナー | ゲーム性・楽しさ・コアサイクルの成立 |
| 2 | テクニカルリード | コード品質・仕様準拠・バグ・パフォーマンス |
| 3 | UIデザイナー | 視認性・操作性・トンマナ |
| 4 | サウンドデザイナー | 音響設計・SE/BGM |
| 5 | アーティスト | ビジュアル品質・世界観 |
| 6 | VFXデザイナー | 演出・アニメーション |

**全ワーカー必須作業:**
1. スクリーンショットを自分の目で視覚確認
2. debug_state.json で内部状態確認
3. 企画書・仕様書を参照
4. フェーズ要件の充足判定（PASS / CONDITIONAL / FAIL）
5. 具体的な問題点と改善提案を出力

### 4.4 Phase 3: ディレクター監査（5名並列）

| # | ディレクター | 監査対象 |
|---|---|---|
| 1 | ゲームデザインリード | プランナーのレビュー結果 |
| 2 | テクニカルディレクター | テクニカルリードのレビュー結果 |
| 3 | アートディレクター | アーティスト・VFX・UIデザイナーのレビュー結果 |
| 4 | サウンドディレクター | サウンドデザイナーのレビュー結果 |
| 5 | QAディレクター | 全ワーカーのレビュー結果（横断） |

**全ディレクター必須作業:**
1. 配下ワーカーのレビュー結果を読む
2. **自分の目で**スクリーンショットを視覚確認（ワーカーの報告を鵜呑みにしない）
3. **自分で**debug_state.json を確認
4. 企画書・仕様書を参照して仕様適合を確認
5. ワーカーのレビューに見落としがないか検証
6. フェーズ要件チェックリストに沿って判定（APPROVE / CONDITIONAL / BLOCK）
7. BLOCK の場合: 具体的な修正指示を出す

### 4.5 Phase 4: 判定ルール

| 状態 | アクション |
|---|---|
| 全ディレクター APPROVE | Phase 5 へ進む |
| CONDITIONAL あり（BLOCK なし） | 条件を明記して Phase 5 へ進む |
| **BLOCK が1件でもあり** | **Phase 5 に進まない。修正して Phase 1 からやり直す** |

### 4.6 Phase 5: プロデューサー報告

Phase 4 で全 APPROVE（または CONDITIONAL のみ）の場合のみ実行。

統合ドキュメント (`docs/reviews/{phase}_testplay_review.md`) を作成:
1. 総合判定テーブル（ワーカー判定 + ディレクター監査判定）
2. フェーズ要件充足マトリクス
3. 各ワーカーの詳細レビュー
4. 各ディレクターの監査コメント
5. 次フェーズに向けたロードマップ

## 5. フェーズ別要件チェックリスト

### 5.1 プリプロ

| 担当 | 要件 | チェック項目 |
|---|---|---|
| プランナー | コンセプト定義 | 企画書が承認済みか。コンセプトが明確か |
| プランナー | コアゲームプレイ詳細定義 | 仕様書でコアサイクルが詳細に定義されているか |
| プログラム | コア部分コーディング | BET→LEVER→STOP×3→判定→IDLE が動作するか |
| プログラム | 技術課題トライ | EXE/APK ビルドが成功するか |
| アート | コンセプトアート | キービジュアルが1枚以上あるか |
| アート | 仮ビジュアル適用 | テキストUIだけでなく、何らかのビジュアルが適用されているか |
| UI/UX | トンマナ定義 | カラーパレットが適用されているか |
| UI/UX | ボタンサイズ・フォント定義 | タッチ操作可能なボタンが実装されているか。フォントが選定されているか |
| UI/UX | 視認性担保 | テキストのコントラストが十分か |
| VFX | 簡易アニメーション | 何らかのアニメーションが動作しているか |
| VFX | 象徴的VFXテスト | コンセプトの核心VFX（花火等）の仮実装があるか |
| サウンド | サウンド選定 | SE/BGMの方向性が決まっているか。仮アセットがあるか |
| サウンド | 技術アイデア | 音響技術の検討がされているか |

## 6. 日常作業のエスカレーション

フェーズレビュー以外の日常作業でもエスカレーションフローを遵守する。

### 6.1 ゲート制パイプライン

```
[企画] → [仕様書] → [全体レビュー] → [実装] → [コードレビュー] → [テスト] → [UIレビュー] → [出荷審査] → [完了]
  G0       G0.5        G0.9           G1        G2             G3        G4          G5           G6
```

| ゲート | ワーカー | ディレクター | エスカレーション |
|---|---|---|---|
| G0 | planner 作成 | — | — |
| G0.5 | planner 作成 | game-design-lead 審査 | APPROVE → G0.9 / BLOCK → G0 |
| G0.9 | 全ワーカー レビュー | 全ディレクター 監査 | 全APPROVE → G1 / BLOCK → 修正 |
| G1 | リードプログラマー + ワーカー | — | — |
| G2 | tech-lead レビュー | tech-director 監査 | APPROVE → G3 / BLOCK → G1 |
| G3 | qa テスト | — | PASS → G4 / FAIL → debugger → G1 |
| G4 | ui-designer レビュー | art-director 監査 | APPROVE → G5 / BLOCK → G1 |
| G5 | — | qa-director 最終審査 | SHIP → G6 / NO SHIP → G1 |
| G6 | — | — | プロデューサーに報告 |

### 6.2 コード変更後の自動パイプライン

```
Step 1: ワーカーレビュー
  tech-lead → コードレビュー
  qa → ビルド + テスト
  ui-designer → スクショ確認

Step 2: ディレクター監査（Step 1 完了後）
  tech-director → tech-lead の結果を監査
  art-director → ui-designer の結果を監査
  qa-director → 全体品質の最終判定

Step 3: 判定
  全 APPROVE → プロデューサーに報告
  BLOCK あり → 修正して Step 1 から
```
